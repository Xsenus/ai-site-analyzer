# План Исправлений Для `ai-site-analyzer`

Дата ревизии: 2026-03-30

Документ относится к роли `ai-site-analyzer` как downstream-сервиса, который выполняет анализ текста сайта, `site-profile` и `ai-search`.

## Цель

Сделать downstream-контракт стабильным и достаточно формализованным, чтобы `ai-integration` мог:
- надежно отличать штатный успех от fallback и от ошибки
- понимать retryability
- не гадать по свободному тексту ошибки

## Что Важно Для Связки Прямо Сейчас

Сейчас основные логические проблемы находятся выше, в `library` и `ai-integration`, но для надежной связки downstream тоже нужно укрепить:
- стабильность схем ответов
- машинные коды ошибок
- различение "нет данных" и "сервис временно недоступен"

## План Исправлений

### 1. Зафиксировать Downstream Contract Для `POST /v1/analyze/json`

Что изменить:
- В `app/api/schemas.py` и `app/api/handlers/analyze_json.py` явно закрепить:
  - success payload
  - validation error payload
  - service error payload
- Для неуспешных ответов вернуть или стандартизовать поля:
  - `error_code`
  - `detail`
  - `retryable`
  - `source`

Почему это важно:
- `ai-integration` должен понимать, когда нужен retry, а когда нужен fallback по ОКВЭД.

Как проверить:
- Пустой `text_par` возвращает предсказуемую validation-ошибку.
- Временная недоступность внешней модели возвращает `retryable: true`.
- Ошибка структуры ответа модели не маскируется под "нет сайта".

### 2. Уточнить Семантику `site_unavailable`

Сейчас fallback по ОКВЭД уже собирается выше, но downstream должен помогать отличать:
- сайт реально недоступен
- сайт доступен, но контент пустой
- сервис анализа временно недоступен
- данные невалидны

Что изменить:
- Привести к одному словарю коды причин для site-related ошибок.
- Если возможно, возвращать machine-readable flags:
  - `site_unavailable`
  - `site_empty`
  - `analysis_unavailable`

Как проверить:
- Один и тот же сценарий всегда возвращает одинаковую классификацию.
- `ai-integration` может выбрать fallback без парсинга текста ошибки.

### 3. Добавить Контрактные Тесты На Стабильность Схем

Что изменить:
- Добавить тесты на `POST /v1/analyze/json`.
- Добавить тесты на `POST /v1/site-profile`.
- Добавить тесты на alias-маршруты `ai-search`, если они участвуют в связке.

Что проверять:
- обязательные поля присутствуют
- структура ответа не меняется случайно
- ошибки имеют ожидаемый формат

### 4. Упростить Диагностику Для Upstream

Что изменить:
- В логах по `analyze/json` и `site-profile` обязательно держать:
  - request id
  - pars_id / company_id, если есть
  - short error class
  - длительность запроса
- По возможности возвращать upstream-friendly diagnostics без утечки лишнего объема текста.

Как проверить:
- По одному request id можно восстановить трассу downstream-вызова.

## Модули, Которые Имеет Смысл Пересмотреть

### `app/api/schemas.py`

- Уточнить и стабилизировать response models.
- При необходимости добавить отдельные модели ошибок.

### `app/api/handlers/analyze_json.py`

- Нормализовать ошибки.
- Зафиксировать machine-readable признаки проблем.
- Добавить тестовые сценарии для разных типов отказа.

### `app/routers/site_profile.py`

- Проверить, что контракт `site-profile` тоже стабилен и диагностируем.

### `app/routers/ai_search.py`

- Если `ai-search` участвует в цепочке downstream-вызовов, зафиксировать формат ошибок и таймаутов.

## Набор Проверок Для `ai-site-analyzer`

### Автотесты

- валидный `analyze/json`
- пустой `text_par`
- невалидный payload
- ошибка вызова LLM или embedding-сервиса
- стабильность response schema

### Ручные Проверки

1. Отправить корректный payload из `ai-integration`.
2. Смоделировать пустой текст.
3. Смоделировать недоступность внешнего AI.
4. Проверить, что все ошибки классифицируются одинаково между повторами.

## Критерии Готовности Сервиса

- `ai-integration` может принимать решения по error-code, а не по строковому сообщению.
- Появляется понятный сигнал для retry и для fallback.
- Схемы ответов покрыты автотестами и не плавают между коммитами.
